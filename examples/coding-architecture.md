# Example: Coding Architecture Decision

Prompt:

```text
这个记忆模块要放进主进程，还是拆成独立 service？
```

Output shape:

## 架构矛盾

主进程集成的优势是调用简单、迭代快、调试成本低。独立 service 的优势是边界清晰、扩展独立、权限和资源隔离更好。

二者的矛盾不是“单体 vs 微服务”的抽象争论，而是“当前变化速度”和“边界稳定程度”的矛盾。

## 阶段判断

如果 API、schema、召回策略、权限模型都还在快速变化，过早拆服务会把不稳定边界固化成长期维护成本。

如果调用量、数据隔离、权限边界、团队边界已经稳定，独立 service 才开始变成主要收益。

## 实践路线

先做进程内模块，但用清晰接口预留服务化边界：

```text
MemoryRepository -> MemoryIndexer -> MemoryRetriever
```

这三个接口分别隔离存储、索引、召回。未来拆服务时，调用方只需要替换接口实现。

## 验证信号

当以下条件同时出现，再拆独立 service：

- QPS 或任务队列开始影响主进程稳定性。
- 记忆模块需要独立扩缩容。
- 权限和数据隔离成为真实需求。
- 接口连续多个迭代保持稳定。
